# pi-git — Build Spec

> Design/implementation spec. Decisions, responsibilities, interfaces, build
> order, acceptance criteria. No function bodies.

**Goal:** Make Pi's message-rewind revert *files too*. Automatically checkpoint the working tree each turn, keyed to the conversation, so when the user rewinds a message (Pi's fork) the file changes made since that message are undone with it — the OpenCode/Claude-Code safety net. Plus git worktrees as an isolation primitive for pi-spawn. **No agent tools — this is pure harness behavior.**

**Architecture:** A thin injectable `git` exec wrapper (checkpoints as commit objects on a private ref namespace; worktree management) wired entirely through **lifecycle hooks** — checkpoint on each turn, restore on fork. The agent never calls pi-git; it just works. One config command for the human.

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, `node:child_process` (or `pi.exec`) for git, `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Application:
- **No tools.** All behavior is hooks; the only user surface is the `/pi-git` config command.
- Emits `git:checkpoint { ref, reason }`, `git:rollback { ref, reason }`. Subscribes `todo:task-complete`.
- Enforcement dial: `off` (disabled) / `notify` (default: checkpoint + restore-on-rewind, quiet in the footer). `block` has no meaning here (nothing to gate) → treated as `notify`. Log `[pi-git]`.

## Design Decisions

1. **Checkpoints as commit objects on a private ref namespace**, not stashes. Snapshot everything (tracked + untracked) without touching the working tree or index: stage into a temporary index (`GIT_INDEX_FILE`), `git add -A`, `git write-tree`, `git commit-tree` → commit sha, store under `refs/pi-git/checkpoints/<entryId>`. Robust where `git stash create` (tracked-only) is not. Restore = read that tree back into the working tree.
2. **Checkpoint each turn, keyed to the user-message entry id.** On `before_agent_start` (fires after the user submits, before the agent loop) create a checkpoint of the tree *as this turn begins* — before the agent's edits for this message. **Entry-id source (confirmed against Pi's types):** the event payload does *not* carry it (`BeforeAgentStartEvent = { prompt, images?, systemPrompt, systemPromptOptions }`), so read it from the session manager — the just-submitted prompt is the current leaf: `ctx.sessionManager.getLeafEntry()`, which is a `SessionMessageEntry { type:"message", id, message: { role:"user" } }`; take `.id`. Guard: if the leaf isn't a `role:"user"` message, scan `getBranch()` backward for the last one. This `id` is the **same id space** as `session_before_fork.entryId` (both `SessionEntry.id`, 8-char hex). Store the checkpoint at `refs/pi-git/checkpoints/<entryId>` — the ref name *is* the entry→checkpoint map, so restore is a direct ref lookup, no side table. Also emit `git:checkpoint`, and optionally `appendEntry` metadata for listing/timestamps. Also checkpoint on `todo:task-complete` (reason `"task complete"`).
3. **Restore on rewind = hook Pi's fork lifecycle (no command).** Pi's message-rewind *is* `/fork` (the `app.session.fork` keybind / Esc-Esc UI); position `"before"` rewinds to just before the selected user message. So:
   - `session_before_fork` → record `{ entryId, position }` in memory (do **not** cancel).
   - `session_shutdown` with `reason === "fork"` → the fork is now committing, so restore the working tree to the checkpoint keyed by that `entryId` (the tree as that turn began, undoing every edit since); emit `git:rollback`. Restoring in `shutdown` rather than `before_fork` avoids reverting files if another extension cancels the fork.
   - Handle `position: "before"` (`/fork`) by restoring the start-of-turn checkpoint. `position: "at"` (`/clone`) keeps the selected message → no restore in v1 (documented).
   The filesystem restore happens during the old instance's shutdown, so it persists into the forked session; the new instance re-inits on `session_start { reason: "fork" }`.
4. **Worktrees** wrap `git worktree add/list/remove` — the isolation primitive **pi-spawn** composes with for parallel agents. Exposed as an *internal capability* (called programmatically by pi-spawn via the event bus / a small API), **not** an agent tool. Governed by config (`worktrees.auto`, `worktrees.baseDir`).
5. **No destructive-command guard.** With every turn checkpointed, `rm -rf src/` is a rewind away, so a guard adds little for local file damage; genuinely irreversible actions (`git push --force`, network) are a different concern that doesn't belong here. Dropped for simplicity.

### Open items for the human
- Default `mode` (recommend `notify`). Whether `snapshotTree` honors `.gitignore` (avoid checkpointing large build artifacts — recommend yes). Worktree base dir. Whether `worktrees.auto` defaults on (recommend off; pi-spawn opts in per call).

## Interfaces / Contracts

```typescript
// git.ts — every method takes an injected exec; returns typed results
interface Worktree { path: string; branch: string; head: string; }
interface Git {
  snapshotTree(): Promise<string>;            // commit sha capturing tracked+untracked (honors .gitignore)
  restoreTree(sha: string): Promise<void>;    // overwrite working tree from a checkpoint sha
  isDirty(): Promise<boolean>;
  updateRef(ref: string, sha: string): Promise<void>;
  readRef(ref: string): Promise<string | null>;
  listRefs(prefix: string): Promise<Array<{ ref: string; sha: string }>>;
  worktreeAdd(branch: string, path: string): Promise<string>;
  worktreeList(): Promise<Worktree[]>;
  worktreeRemove(path: string): Promise<void>;
}
function createGit(exec: ExecFn, cwd: string): Git;

// checkpoints.ts — keyed by the user-message entry id (Decision 2/3)
interface Checkpoint { entryId: string; ref: string; sha: string; reason: string; createdAtIso: string; }
function currentUserEntryId(sm: ReadonlySessionManager): string | null;      // getLeafEntry() → last role:"user" fallback
function checkpoint(git: Git, entryId: string, reason: string, now: string): Promise<Checkpoint>;  // ref = refs/pi-git/checkpoints/<entryId>
function findCheckpoint(git: Git, entryId: string): Promise<Checkpoint | null>;   // just readRef(refs/pi-git/checkpoints/<entryId>)
function restoreTo(git: Git, cp: Checkpoint): Promise<void>;

// worktrees.ts — pi-spawn's isolation seam (internal capability, not a tool)
function createWorktree(git: Git, baseDir: string, name: string): Promise<Worktree>;
function removeWorktree(git: Git, path: string): Promise<void>;

// NO tools. events: git:checkpoint {ref, reason}, git:rollback {ref, reason}
// config: { mode, checkpoint: { include: "all" | "tracked" }, worktrees: { auto: boolean, baseDir: string } }
```

## Build Sequence

- [ ] **1 · Scaffold** — per house-style. Accept: `bun test` runs.
- [ ] **2 · `git.ts` snapshot/restore** — Accept (integration test in a temp repo): `snapshotTree` then dirty the tree then `restoreTree` returns the exact prior content, **including a previously-untracked file**, and the user's real index is untouched.
- [ ] **3 · Checkpoints** — Accept: `checkpoint` writes `refs/pi-git/checkpoints/<entryId>` + persists metadata; `findCheckpoint(entryId)` reads it back; `restoreTo` round-trips (temp repo).
- [ ] **4 · Worktrees** — Accept (temp repo): `createWorktree` makes a linked worktree on a new branch under `baseDir`; `worktreeList` includes it; `removeWorktree` removes it.
- [ ] **5 · Checkpoint hook** — Accept: `before_agent_start` in `notify` resolves the user-entry id via `currentUserEntryId(ctx.sessionManager)` (leaf `role:"user"` entry, with the `getBranch()` backward-scan fallback), checkpoints to `refs/pi-git/checkpoints/<entryId>`, and emits `git:checkpoint`; `todo:task-complete` adds one; `off` does nothing.
- [ ] **6 · Rewind hook** — Accept: a simulated `session_before_fork { entryId, position:"before" }` followed by `session_shutdown { reason:"fork" }` restores the working tree to that entry's checkpoint and emits `git:rollback`; a `position:"at"` fork does not restore; a *cancelled* fork (no shutdown) does not restore.
- [ ] **7 · Config** — Accept: `/pi-git` views/sets `mode`, `checkpoint.include`, `worktrees.auto`/`baseDir`, persisted like the other extensions' config file (`~/.pi/agent/pi-git.json`).

## Test Strategy

`git.ts`/checkpoints/worktrees — integration tests against a temp git repo in a tmp dir (real git, isolated). The fork-restore *selection* logic (which checkpoint for which entry/position, capture-then-restore ordering) — pure unit tests with a fake `Git` + captured events. No agent tools to test.

## Risks / Notes

- Temporary-index snapshotting (`GIT_INDEX_FILE`) must not disturb the user's real index — verify in the step-2 integration test.
- Entry-id sourcing is settled (Decision 2): read the leaf from `ctx.sessionManager.getLeafEntry()`, same id space as `session_before_fork.entryId`. The only live-Pi check left is confirming the user prompt is already the leaf when `before_agent_start` fires (the backward-scan fallback covers it if not).
- Restoring the tree on fork throws away edits made after the rewind point — that is the intended semantics (it mirrors the conversation rewind), but note it clearly since it is destructive to post-rewind work.
- Not a git repo? Degrade to a no-op with a logged `[pi-git]` notice; never hard-fail a turn because the cwd isn't a repo.
